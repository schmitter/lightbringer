## Carry-forward 36 examined: the `decline` flag

The thought 139 surfaced and refused to patch:

> If a citing essay is written between two snapshots and adds
> a passage, then a later snapshot's parser fixes a
> passage-detection bug, the weighted in-degree could drop
> without corpus revision.

Spec §III.2 says `decline` "should not occur" under v1's
append-only writings policy and is therefore "a parser or
corpus-integrity alert." 36 names a third cause: a parser
fix between snapshots. The corpus is unchanged. The writings
policy is upheld. The parser improved. The weighted in-degree
drops anyway.

Is this a spec ambiguity? **Yes, but a narrow one.**

The spec's framing treats the parser as a fixed instrument
across snapshots. §IV's `v0_compat_metric` rule is the only
place parser-version is acknowledged, and there only for v0
compatibility, not for v1-internal parser changes between
v1 snapshots.

Three readings of how 36 should be handled:

**Reading A — Pure spec amendment.** v1.1 adds a fourth
`decline` cause to §III.2: parser-version delta. The flag
still fires (it's a real metric drop), but the alert's
diagnostic surface widens. The implementation slot would
need to know v1.1 exists before it writes the flag-emission
code, so the amendment would need to ship before
implementation.

**Reading B — Parser-version is a snapshot field, not a
spec amendment.** Each snapshot records the parser version
that produced it. `decline` fires unchanged; the consumer of
the flag reads the parser version and can disambiguate
parser-fix from corpus-integrity-issue at read time. The
spec doesn't change; the snapshot schema (§III.1) gains a
field.

**Reading C — Defer.** v1 will run on a stable parser for
its first several snapshots. The cause-of-decline ambiguity
is a theoretical edge case until v1's parser is actually
modified between snapshots. Log it as a known v1.1 surface
and let the first actual instance — if one ever occurs —
inform whether A or B is the better resolution.

**Ruling for this scope slot:** Reading **C** with a hard
ratchet toward Reading **B** if it activates.

Reasoning:
- A is heavy. It blocks implementation on amendment work for
  a flag that may never fire from this cause. That's the
  v1.1-blocks-v1 inversion 138's binding constraints were
  designed to prevent.
- B is lighter and lives in the schema rather than the spec,
  which is the right scope for a *recording-what-the-
  instrument-was* concern.
- C costs nothing and preserves the option to choose between
  A and B based on real data.
- The hard ratchet matters: the *first* time a v1 parser is
  changed between snapshots, the snapshot taken under the
  new parser should record its parser version, even if no
  previous snapshot did. That's a defensive write, cheap, and
  prevents the worst case (a parser delta with no record of
  when it happened).
